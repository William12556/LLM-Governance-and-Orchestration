## 5.0 Behavior

```yaml
behavior:
  expected: >
    Worker completes all specified file modifications within phase_max_iterations.
    Each file is read once, edited once (batching multiple changes per file into
    a single edit call), and not re-read after a successful edit.
  actual: >
    Three failure modes observed in log analysis:

    1. REDUNDANT TOOL CALLS PER ITERATION: Model issues identical tool calls
       3-5 times in a single model response (e.g. same edit call issued 4 times
       at phase iteration 16, same is_setup_complete edit issued 5 times at
       iteration 19). Only the first succeeds; subsequent calls fail silently
       or produce no-ops. Each group of redundant calls is contained within
       one phase iteration but wastes model context.

    2. EXCESSIVE RE-READS: app.py was read 9 times across 20 iterations
       (duplicate read warnings at counts 2-9). The recipe instructs
       "Do not re-read files you have already read unless necessary" but
       the model re-reads after every edit to confirm state rather than
       trusting the edit tool result.

    3. SINGLE-ENTRY EDITS: Model issues one edit entry per call even when
       multiple changes to the same file are required in sequence. This
       multiplies the number of edit+read cycles required per file.
  impact: >
    phase_max_iterations exhausted before task completes. Loop exits rc=1.
    Reviewer never runs. Task requires re-run with reset, adding human
    overhead and risk of partial-state conflicts.
  workaround: "Raise phase_max_iterations to 50 in config.yaml as a palliative measure."
```

## 6.0 Analysis

```yaml
analysis:
  root_cause: >
    The ralph-work.yaml recipe provides no tool-specific usage guidance beyond
    a bare tool list injected via {{TOOLS}}. The injected list contains only
    tool names and parameter signatures — no information about:
      - Whether edits array supports multiple entries per call (it does)
      - When to prefer write over edit (full file replacements)
      - That re-reading after a successful edit is unnecessary
      - That repeating a successful tool call is wasteful

    Without this guidance, Devstral-Small exhibits three compensatory
    behaviours: defensive re-issue of calls it is uncertain succeeded,
    re-read confirmation after each edit, and single-entry edit calls
    requiring multiple round-trips per file.

  technical_notes: >
    The {{TOOLS}} injection is generated by format_tool_signatures() in
    orchestrator.py, which renders parameter names and types only. This is
    correct for preventing hallucinated argument names but insufficient for
    teaching the model how to use the tools efficiently.

    The recipe already contains "Do not re-read files you have already read
    unless necessary" but this is insufficiently specific — it does not tie
    the constraint to the edit success confirmation pattern.

    Three targeted additions to ralph-work.yaml address all three failure modes
    without orchestrator changes:
      A. TOOL DISCIPLINE block: explicit constraints against redundant calls,
         re-read confirmation, and single-entry batching
      B. Tool usage notes after {{TOOLS}}: per-tool guidance (edit batching,
         write for rewrites, trust edit result)
      C. Edit success recognition: "Successfully applied" in result = success,
         do not re-issue

  related_issues:
    - issue_ref: ""
      relationship: ""
```

## 7.0 Resolution

```yaml
resolution:
  assigned_to: ""
  target_date: ""
  approach: >
    Modify framework/ai/ael/recipes/ralph-work.yaml:

    CHANGE 1 — Add TOOL DISCIPLINE block to CONSTRAINTS section:
      - Do not repeat a tool call that has already succeeded this iteration.
        A successful edit result contains "Successfully applied" — trust it, move on.
      - Batch all changes to the same file into a single edit call with multiple
        entries in the edits array. Do not issue sequential single-entry edit calls
        to the same file.
      - Use write (not edit) when replacing an entire file.
      - Do not re-read a file after a successful edit. The diff in the tool result
        confirms the change. Re-read only if the edit failed.

    CHANGE 2 — Add tool usage notes after {{TOOLS}} injection:
      TOOL USAGE NOTES:
      - edit: edits array accepts multiple entries — batch all changes to one
        file in a single call. Do not re-read after success.
      - write: use for new files or full file replacements.
      - read: read a file once before editing; do not re-read after successful edit.
      - grep: use to locate content before editing when line text is uncertain.

  change_ref: ""
  resolved_date: "2026-03-27"
  resolved_by: "William Watson"
  fix_description: >
    Added TOOL DISCIPLINE block to CONSTRAINTS section in ralph-work.yaml.
    Added TOOL USAGE NOTES after {{TOOLS}} injection. Removed bare re-read
    constraint (subsumed by TOOL DISCIPLINE). Version bumped to 1.2.0.
    Applied to framework, skel, GTach, e-Paper-IP-Display, solax-modbus.
```
